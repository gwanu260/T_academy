# Form 관련 백그라운드 정보
    - Client가 Server 용어 정의
        - Client : 일반 유저, 브라우저를 통해서 소통(PC, 모바일웹)
        - Server : 인프라구성 + (*)실제 서비스 개발
            - 물리적 : s/w로 구성된 프로그램이 작동되는 공간
                - 하드웨어(CPU, ram, 디스크, 네트워트크장비,..)
                    - 관리(사내, 온프라(레)미스) => IDC센터 (위치, 임대) => AWS같은 클라우드 회사 임대(클라우드)
            - S/W : 
                - 서버운영에 관련된 s/w : os, docker, 네트워크(VPC),.....
                - (*)클라이언트에서 서비스를 제공하는 s/w
                    - fastapi기반으로 개발한 s/w => 웹서비스  
                    - springboot기반으로 개발한 s/w => 웹서비스 : 서버 개발
    
    - client가 server(s/w) 에게 데이터를 전달하는 방법
        - 방식 => method => get(헤더로 데이터를 전송) vs post, put, delete, ...(바디로 데이터를 전송)
        - 코드 관점
            - html
                - <form>
                    - get, post, put, delete 방식 모두 가능함
                    - form(폼) 전송 -> 특이점(화면이 껌벅댐->화면 새로그림)
                    - method 지정 않하면 -> get방식
                    - action 지정 않하면 -> URL은 현재 페이지의 URL
                - <a>
                    - get 방식만 존재
                    - 링크
            - js
                - 오리지널 api -> 무시 -> 너무 복잡
                - (*)fetch()
                    - html5에서 공식 지원하는 (ECMA에서 추가함) 통신 함수
                    - 모든 통신 방식(method)을 지원
                    - ajax(아작스,에이작스) 통신 지원
                        - 화면 껌벅임 X
                        - 백그라운드(뒤로 통신), 비동기(화면 렉 X) 통신
                            - 지도 서비스, 검색창(추천 검색어 노출)
                        - 주고 받는 데이터의 형태 = XML, (*)json, 텍스트등
                    - 프로그래밍 적으로 원하는 타이밍에 통신 진행 스타일
                    - 대시보드상 
                        - 차트 드로잉
                        - 챗봇, 채팅
                        - 검색                        
                - 기타 - 외부 라이브러리 - (*)axios등 제품 존재함


# 템플릿 엔진
    - 목적
        - 개발자 코드와 디자인너 코드가 겹치는 것을 분리하기 위해서 탄생.
            - 과거 : html 안에 프로그램 코드(java, cgi,...) 포함되어 있었음
            - 템플릿 엔진 개발하여 분리
                - java(or jps) + xslt(템플릿)
                    - 현재 : java + 타임리프(템플릿)
                - python + jinji2(템플릿)
        - 특징
            - 디자이너가 약간의 프로그래밍(변수, 함수호출, 흐름제어, 주석)을 할수 있게 제공한다!!
    - 값 출력
        - {{ 값(변수|연산식|함수호출) }}
    - 흐름제어
        - {% for ~ %}
        - {% endfor %}
        - ...
    - 주석
        - {# #}
        - 클라이언트에게 개발자간의 대화 내용(중요 주석)을 숨김다
    - html 내부에 다른 html 포함
        - {% include "xxx.html" %}
        - 거대한 html을 기능별로 분리하여서 필요시, 조립하여 페이지 구성
    - 상속
        - 자주 사용하는 페이지의 기본 구조, 표현등을 미리 정의하여, 상속하여 구성
        - 기본뼈대 : base.html
            - 개별페이지 구성
                - base.html -> extends -> 커스텀 구성 
                    => sales_board.html, sales_detail.html, ...
                - {% extends "base.html" %}
            - 커스텀 구성
                - {% block 블럭명 %}
                - {% endblock %}
        - Human -> 상속 -> Human
                        -> XMan (사람인데 특정 능력치 커스텀) -> XHan2

# 세션 및 쿠키
    - Session
        - 서버측이 특정 사용자(브라우저)를 기억|구분하는 방법
        - http의 특징 
            - stateless => 직전 접속한 브라우저(사용자) 기억 x
            - 모든 요청은 새로운 유저이다.
        - 사용자 정보(설정된 값, 아이디등)를 저장 (거의 자동으로 처리됨)
            - 사용자별 id(해시값->중복X)를 생성
            - 서버측 메모리 or 스토리지(redis등 NOSQL S/W)등에 저장
            - 쿠키(저장공간, 이동공간)에 담아서 브라우저로 전송
            - 브라우저에 저장
            - 요청할때마다 브라우저에 저장된 id값을 가지고, 서버측으로 요청 진행
            - 쿠키에서 전달된 ID와 서버측에 저장된 ID를 비교하여 동일 사용자임을 인식
        - 은행등 보안에 민감한 사이트는 => 시간제약을 부여(10분이후 자동 로그아웃등등)
        - 키:값
    - 쿠키
        - 클라이언트측 브라워저의 저장되는 정보
        - 서버가 보낸 데이터를 브라우저에 저장하는 방식
        - 키:값
    - FASTAPI
        - SessionMiddleware 클레스 활용
        - starlette 패키지하위 위치함

# Pydantic (파이단틱)
    - 목적
        - 데이터 검증(타입, 길이 => 유효성검사)
        - 데이터의 직렬화 처리 =>JSON 변환이 쉬움
            - 파이썬 dict 자료구조를 => 문자열(텍스트)로 변환(JSON) : 직렬화
            - JSON 형태로 구성된 문자열 => dict로 자료구조로 변경 : 역직렬화
        - 직렬화 이유 => 자료구조를 문자열로 변환해서 => 통신시 데이터 전송에 사용 -> 간단하게 json 형태로 데이터를 묘사하고 전달 가능함
        
        - 개별적 데이터를 하나의 구조형 형태(JSON)로 구성하여 통신가능.
        - 파이썬에서는 클래스와 객체에 대한 묘사나 사용이 원할해야 이용 가능함.
    - client <-> json (형태로) <->server
